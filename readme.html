Introduction

openDMR is a cross-platform C++ Digital Mobile Radio (DMR) master/server that authenticates Homebrew nodes, routes group and private traffic, supports static TS1/TS2 subscriptions, and optionally adds APRS, hotspot location capture, DMR-SMS, SQLite logging, a multi-page embedded HTTP UI, and multi-peer uplinks/OpenBridge bridging.

Highlights: UDP Homebrew networking, SHA-256 repeater auth, static talkgroup config via RPTC, optional hotspot LAT/LON capture, APRS/SMS modules, SQLite logging, embedded dashboard/monitor plus Hotspot/Map pages, classic OpenBridge peers, and optional unified uplinks for Homebrew/OpenBridge backbones.

Features

  • DMR UDP protocol handling for DMRD, RPTL, RPTK, RPTC/RPTO, RPTPING, and FMRPING
  • Talkgroup routing with owner-slot arbitration, scanner TG fan-out, and per-slot static TS1/TS2 subscriptions
  • Private-call forwarding to the last seen destination slot plus Parrot echo playback on TG 9990
  • SHA-256 challenge/response auth with CSV-backed per-node passwords and configurable unknown-ID policy
  • Optional APRS heard reporting plus APRS-IS position parsing, DMR-ID to callsign lookup, reconnect/keepalive handling, and merged APRS/hotspot map data
  • Optional DMR-SMS buffering with TG allow-lists, private-call policy, frame/time limits, and UDP emission to external tools
  • Optional parallel DMRD receive workers via WorkerThreads (1..32), with stream/node sharding to preserve packet affinity
  • Optional SQLite3 activity database with recent log, active-call, and online status views
  • Embedded HTTP monitor with static file serving plus JSON/text endpoints for config, auth, log, active calls, APRS/map data, OpenBridge status, and raw /STAT
  • Additional monitor JSON for static TS1/TS2 visibility via /api/systemstg and merged APRS/hotspot positions via /api/aprs, alongside richer log grouping modes for recent activity
  • Self-service web registration, login, logout, and profile updates backed by auth_users.csv and the configured DMRIds.dat file
  • Modern bundled web UI: Dashboard, Monitor, OpenBridge, Hotspot, Map, DV Manager, Register, and Profile
  • Classic multi-peer OpenBridge mode with up to 3 peers, TG filters, DNS refresh, HBLink framing, optional enhanced HMAC, and relaxed checks
  • When built with USE_UPLINK: up to 8 generic uplinks plus 3 OpenBridge-class uplinks, including Homebrew login/auth/ping handling and upstream static TS1/TS2 configuration
  • Radio blacklist support, DMR-ID profile lookups, bundled INI parsing, and Linux/Windows portability

Quick Start

Linux (full feature build with unified uplinks)

Copyg++ server.cpp -o server   -DUSE_SQLITE3 -DUSE_OPENSSL -DHAVE_APRS -DHAVE_SMS -DHAVE_HTTPMODE -DUSE_UPLINK   -lsqlite3 -lcrypto -lssl -lpthread
./server

Linux (classic build without USE_UPLINK)

Copyg++ server.cpp -o server   -DUSE_SQLITE3 -DUSE_OPENSSL -DHAVE_APRS -DHAVE_SMS -DHAVE_HTTPMODE   -lsqlite3 -lcrypto -lssl -lpthread
./server

Windows (MinGW)

Copybuild_mingw.bat
content\server.exe
Note: Run the binary from the runtime directory containing dmr.conf, talkgroup.dat, and banned.dat. auth_users.csv and DMRIds.dat are required when [Auth] Enable=1, and the HTTP monitor expects the configured [Monitor] Root directory to contain the bundled web assets.

Architecture

The core is an event-driven UDP loop with light worker threads for timing and parrot playback; APRS, hotspot location snapshots, SMS, OpenBridge, and the embedded HTTP monitor are optional sub-systems layered around the same server state. Current builds also maintain an APRS position cache and expose merged hotspot/APRS snapshots to the bundled map UI.

Build & Install

Dependencies

  • C++17 compiler
  • POSIX sockets (Linux) or WinSock (Windows)
  • (Optional) SQLite3 for /api/log, /api/active, and persistent activity state
  • (Optional) OpenSSL for enhanced OpenBridge HMAC helpers
  • (Browser runtime) the bundled map page loads Leaflet/OpenStreetMap tiles from public CDNs/services

Compile

g++ -O2 -std=c++17 server.cpp -o dmr -lpthread

With common optional modules:

g++ -O2 -std=c++17 server.cpp -o dmr   -DUSE_SQLITE3 -DUSE_OPENSSL -DHAVE_APRS -DHAVE_SMS -DHAVE_HTTPMODE   -lsqlite3 -lcrypto -lssl -lpthread

Enable the unified uplink engine:

g++ -O2 -std=c++17 server.cpp -o dmr   -DUSE_SQLITE3 -DUSE_OPENSSL -DHAVE_APRS -DHAVE_SMS -DHAVE_HTTPMODE -DUSE_UPLINK   -lsqlite3 -lcrypto -lssl -lpthread

DMR Protocol Format

This section documents the subset handled by openDMR based on the server implementation.

DMRD (55 bytes, no HMAC)

OffsetSizeFieldNotes
04MagicASCII "DMRD"
41Flags (internal)See Flags below
53Radio IDget3(pk+5)
83DestinationGroup or private ID (TG or radio), get3(pk+8)
114Node IDget4(pk+11)
151FlagsStart/End, Private, Slot bit
164Stream IDget4(pk+16)
2035PayloadDMR voice/data block (51 bytes total incl. header subset)
…—Total55 bytes

Flags

  • bStartStream: (flags & 0x23) == 0x21
  • bEndStream: (flags & 0x23) == 0x22
  • bPrivateCall: (flags & 0x40) != 0
  • Slot select: bit 0x80 is set for TS2 in forwarding, cleared for TS1

Login / Control Frames

TypeMagicSizeKey FieldsDescription
Login"RPTL"8NodeIDNode requests login; server replies RPTACK + salt
Auth"RPTK"40NodeID, SHA256(salt||pass)Client sends auth response; server replies RPTACK or MSTNAK
Config"RPTC"/"RPTO"≥12NodeID, INI-like payloadApplies static TGs via TS1=/TS2= (CSV) and may also carry hotspot coordinates via LAT/LATITUDE plus LON/LNG/LONG/LONGITUDE
Ping"RPTPING"11NodeIDPong with "MSTPONG" + NodeID
Ping2"FMRPING"11—Compatibility pong "FMRPONG"
Logout"RPTCL"9NodeIDNode disconnect
Status"/STAT"5—Local status dump to 127.0.0.1:Port

Special Talkgroups

  • Unsubscribe all: 4000 (on start, server clears subscriptions)
  • Parrot: 9990 (records on start, replays on end)
  • APRS trigger: 900999 (heard report)
  • Scanner: 777 (temporary owner until traffic ends)

Packet Types & Flow

  1. Login: RPTL → RPTACK(salt)
  2. Auth: RPTK(SHA256(salt||pass)) → RPTACK or MSTNAK
  3. Config: RPTC/RPTO sets static TGs for TS1/TS2 and can also publish hotspot coordinates
  4. Traffic: DMRD frames → routed to all subscribers of the talkgroup (owner slot arbitration)
  5. Keepalive: RPTPING ↔ MSTPONG
  6. Logout: RPTCL

Threading & Timing

  • Time thread (time_thread_proc): increments g_tick every 50ms and g_sec each second.
  • RX worker pool (start_rx_workers, optional): when [Server] WorkerThreads > 1, 55-byte DMRD frames are sharded by stream ID (or node ID fallback) onto a fixed worker queue so related traffic stays on the same worker.
  • Main loop: select() watches the primary UDP socket plus configured uplink/OpenBridge sockets; non-worker traffic is processed inline under the shared state lock.
  • HTTP/UI data prep: APRS position caches and hotspot snapshots are reused by /api/aprs and /api/systemstg so the bundled map and hotspot pages can render live state without touching the SQLite log path.
  • Parrot playback (parrot_playback_thread_proc): replays buffered frames at roughly 20ms pacing after the end of a Parrot stream.
  • Housekeeping: APRS reconnect/keepalive, SMS buffer expiry/emission, auth reload checks, and stale SQLite active-row cleanup run in the main runtime loop.

Core Data Structures

  • struct node: nodeid, dmrid, salt, sockaddr, hitsec, slots[2], auth flag, timer, static TGs, and optional hotspot latitude/longitude
  • struct slot: back-ref node, slotid, current tg, parrot buffer/seq/timestamps, optional SMS buffer
  • struct talkgroup: tg number, ownerslot, tick, linked-list subscribers
  • class memfile: in-memory expanding buffer (Read/Write/Seek)
  • class config_file: INI loader (getstring, getint)

State & Flow

Owner-slot arbitration prevents collisions: the first slot to key up on a TG takes ownership; ownership drops on end/timeout (~1.5s).

  • Joining a TG calls subscribe_to_group() (singly linked list insert).
  • Leaving TG or on control TG 4000 calls unsubscribe_from_group().
  • Scanner owner (g_scanner) follows the last active slot until end/timeout.

Uplinks & OpenBridge

openDMR supports two bridging models. In classic builds it loads up to three [OpenBridge1..3] peers. When compiled with USE_UPLINK, it switches to a unified uplink engine that can manage up to eight generic uplinks plus three OpenBridge-class entries through the same forwarding loop.

Classic / OpenBridge capabilities

  • Up to 3 OpenBridge peers in classic builds
  • Per-peer TG allow-list with PermitAll / PermitTGs
  • Friendly aliases surfaced in logs and the bundled OpenBridge page
  • Periodic DNS re-resolution via ResolveInterval
  • Legacy or HBLink-compatible framing
  • Optional enhanced HMAC validation/signing when built with USE_OPENSSL

Unified uplink additions (USE_UPLINK)

  • Up to 8 [Uplink1..8] entries plus 3 [OpenBridge1..3] entries
  • Homebrew uplinks perform RPTL/RPTK login, periodic ping, and optional static TS1/TS2 config pushes
  • OpenBridge uplinks share the same loop-prevention ring buffer and TG filtering policy
  • Type=brandmeister or Type=tgif automatically implies the Homebrew protocol when Protocol is omitted
  • The bundled /api/openbridge endpoint reports OpenBridge-class peers only; there is no dedicated JSON endpoint for generic Homebrew uplinks yet

DV Manager & Browser Tools

The packaged www folder now includes a browser-side utility page at dvmanager.html, alongside the newer hotspot.html and map.html monitoring pages. It is separate from the server JSON API: the page downloads public contact datasets directly from the browser, lets you filter records by country or undefined country, and exports device-specific CSV/TXT contact files without writing anything back to the server.

Included browser tools

  • Public dataset downloads from radioid.net users/repeaters and freeradioid.net users
  • Country-based filtering, “select all”, and “undefined country only” filtering
  • Device exports for Ailunce HD1, Motorola, RT52, RT3S/RT90/RT84/RT82, AnyTone, GD-77, TYT, Baofeng DM-1701, DVPi, PI-Star, Nextion, BlueDV, and Abbree DM-F8
  • In-browser APRS-IS passcode generation from a callsign

Front-end runtime additions

  • app.js centralizes theme persistence, auth token storage, polling, and page rendering helpers
  • styles.css provides the shared modern dashboard layout, cards, tables, chips, map styling, and light/dark themes
  • The Login/Profile flow uses localStorage for browser persistence and sends the token in X-Auth-Token unless a header is already supplied
  • app.js also adds pagination for hotspot/APRS tables, callsign search for the map table, and Leaflet marker rendering for merged APRS/hotspot positions
  • Dashboard, Monitor, OpenBridge, Hotspot, Map, Register, and Profile all auto-refresh about every 3 seconds
Important: DV Manager depends on the browser being able to reach the public JSON sources. It does not consume /api/log, /api/active, or other embedded monitor endpoints.

Hotspot & APRS Map

Current builds bundle two additional monitoring pages on top of the dashboard/monitor/OpenBridge views. hotspot.html focuses on static talkgroup announcements received from repeater config frames, while map.html merges APRS station positions with hotspot coordinates learned from RPTC/RPTO payloads.

hotspot.html

  • Backed by /api/systemstg
  • Shows DMR ID, callsign, node ID, auth state, current TS1/TS2, configured static TS1/TS2 CSVs, and last-seen age
  • Includes summary metrics for hotspot count, unique static TGs, and total TS1/TS2 assignments
  • Paginates large hotspot lists in the browser via app.js

map.html

  • Backed by /api/aprs and rendered with Leaflet plus OpenStreetMap tiles
  • Merges APRS station packets with hotspot coordinates so both can appear on the same map
  • Provides callsign/DMR-ID/node search, page navigation, popup details, and map centering from the results table
  • Exposes APRS station count, hotspot count, total markers, and latest-seen summary cards
Hotspot coordinates: when a repeater sends RPTC/RPTO, the server now accepts LAT/LATITUDE and LON/LNG/LONG/LONGITUDE fields in addition to TS1/TS2. Those coordinates are surfaced through /api/aprs and used by map.html.

Web Monitor & API

When compiled with HAVE_HTTPMODE and enabled in [Monitor], openDMR serves a static web UI plus JSON/text endpoints from the configured document root. The embedded HTTP server enforces explicit client limits and timeouts, while sessions are stored in memory only.

Included pages

  • index.html: dashboard with active calls, recent log data, summary metrics, and local/APRS/SMS/OpenBridge badges
  • monitor.html: parsed table view of /api/stat plus collapsible raw /STAT output
  • openbridge.html: OpenBridge peer health, status, timers, filters, and enhanced/HBLink indicators
  • hotspot.html: hotspot/static-talkgroup table with TS1/TS2 metrics and per-node visibility from /api/systemstg
  • map.html: Leaflet APRS/hotspot map with merged positions from /api/aprs
  • dvmanager.html: browser-side contact export and APRS passcode tool
  • register.html: self-service account registration
  • profile.html: authenticated profile and password update page

Authentication headers

Authenticated requests use a token in X-Auth-Token. The server also accepts the Authorization header as a fallback. Tokens are stored only in memory, refreshed on each valid request, and expire after 24 hours.

Static serving: requests that do not hit an /api/* route are resolved against the configured document root. / falls back to index.html.

On startup the server loads talkgroup.dat and banned.dat, normalizes the SQLite schema when logging is enabled (including SRC/SEQ migration, index creation, and resetting stale ACTIVE/CONNECT flags), then opens the main UDP socket plus any configured uplink/OpenBridge sockets. When WorkerThreads is greater than 1, the receive worker pool is started before entering the main event loop. The bundled front-end polls the main JSON endpoints roughly every 3 seconds, persists theme/login state in localStorage, and now also drives hotspot/APRS pagination plus live map rendering through /api/systemstg and /api/aprs.

Logging

Console logs use colored levels via logmsg(level, timed, ...) (Linux ANSI / Windows Console API). SQLite logging writes to a LOG table with columns DATE, RADIO, TG, TIME, SLOT, NODE, ACTIVE, CONNECT, SRC, SEQ. On startup, current builds backfill missing SRC/SEQ columns when needed, seed SEQ from existing row IDs, reset stale ACTIVE/CONNECT flags, and create helper indexes for active and grouped monitor queries. The embedded monitor further enriches rows with callsign lookups from DMRIds.dat plus recent-source markers (local vs. OpenBridge, APRS, SMS) so the dashboard can render richer status badges.

FAQ

What happens when a node sends to an unknown TG?

The server does not create arbitrary talkgroups on demand during a normal key-up. Groups must already exist from talkgroup.dat or from static TS1/TS2 configuration; otherwise the server logs a key-up on a non-existent group and unsubscribes the slot.

How are private calls routed?

If the destination is a radio ID within range and present in the node index, the frame is forwarded to that radio's last seen slot, with the slot bit adjusted. SMS forwarding can also allow private messages when AllowPrivate=1.

What does the embedded web server need?

Build with HAVE_HTTPMODE, set [Monitor] Enable=1, and make sure the configured Root directory contains the bundled index.html, monitor.html, openbridge.html, hotspot.html, map.html, dvmanager.html, app.js, and styles.css assets. The map page additionally expects browser access to Leaflet/OpenStreetMap tile resources.

How does registration/profile storage work?

Registration adds entries to both the auth CSV and the configured DMRIds.dat file. Login returns an in-memory session token, profile GET/POST uses that token, and profile updates can change the display name and password for an authenticated DMR-ID.

What changes when I build with USE_UPLINK?

The server switches from the classic OpenBridge-only loader to the unified uplink engine. That adds [Uplink1..8] support, Homebrew upstream login/auth/ping handling, upstream static TS1/TS2 pushes, and still keeps [OpenBridge1..3] available for OpenBridge-class links.

How does Parrot work?

On start: create a memfile buffer; on end: spawn a playback thread to echo frames back at roughly 20 ms cadence after a short delay.

Does DV Manager need the server database?

No. dvmanager.html downloads public JSON datasets directly in the browser and exports files locally. It is packaged with openDMR for convenience, but it is not backed by the embedded SQLite log or the monitor JSON API.

How do the Hotspot and Map pages get their data?

hotspot.html uses /api/systemstg for live static TS1/TS2 data per node, while map.html uses /api/aprs to merge APRS station positions with hotspot coordinates captured from repeater config frames.

How can a hotspot publish its position?

Include coordinates in the repeater config payload using LAT or LATITUDE plus one of LON, LNG, LONG, or LONGITUDE. The server stores those values on the node, surfaces them through /api/aprs, and shows them on map.html when they are valid.
