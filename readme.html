Introduction

openDMR is a cross-platform C++ Digital Mobile Radio (DMR) master/server that links repeaters, routes talkgroups, authenticates nodes, and optionally integrates APRS, DMR-SMS, and SQLite logging. It is built around a compact core with a focus on performance and clarity.

Highlights: UDP networking, SHA-256 authentication, Parrot echo TG, APRS/SMS modules, SQLite logging, embedded web dashboard/monitor, OpenBridge peers, Linux/Windows support.

Features

  • DMR UDP protocol handling (DMRD, RPTL, RPTK, RPTC, heartbeats)
  • Talkgroup management with owner-slot arbitration & scanner TG
  • Parrot TG (9990) recording and playback
  • SHA-256 challenge/response auth with per-node policy
  • Optional APRS: heard reports to APRS-IS
  • Optional DMR-SMS: frame buffering and UDP emission
  • Optional SQLite3: activity & transmission logs
  • Embedded HTTP monitor with dashboard, live status, and JSON API
  • Self-service web registration, login, and profile updates backed by auth and DMR IDs files
  • Up to 3 OpenBridge peers with TG filters, aliasing, DNS refresh, and optional enhanced HMAC
  • Radio blacklist support and DMRIds name/callsign lookup
  • INI configuration parser included
  • Cross-platform (Linux / Windows)

Quick Start

Linux (full feature build)

Copymake
./server

Linux (manual compile)

Copyg++ server.cpp -o server   -DUSE_SQLITE3 -DUSE_OPENSSL -DHAVE_APRS -DHAVE_SMS -DHAVE_HTTPMODE   -lsqlite3 -lcrypto -lssl -lpthread
./server

Windows (MinGW)

Copybuild_mingw.bat
content\server.exe
Note: The current package ships a web UI under content/www and expects dmr.conf, auth_users.csv, talkgroup.dat, banned.dat, and optional www/DMRIds.dat in the runtime directory.

Architecture

The core is an event-driven UDP loop with light worker threads for timing and parrot playback; APRS, SMS, OpenBridge, and the embedded HTTP monitor are optional sub-systems layered around the same server state.

Build & Install

Dependencies

  • C++17 compiler
  • POSIX sockets (Linux) or WinSock (Windows)
  • (Optional) SQLite3
  • (Optional) OpenSSL (if you wire HMAC helpers)

Compile

g++ -O2 -std=c++17 server.cpp -o dmr -lpthread

With optional modules:

g++ -O2 -std=c++17 server.cpp -o dmr -DUSE_SQLITE3 -DHAVE_APRS -DHAVE_SMS -lsqlite3 -lpthread

DMR Protocol Format

This section documents the subset handled by openDMR based on the server implementation.

DMRD (55 bytes, no HMAC)

OffsetSizeFieldNotes
04MagicASCII "DMRD"
41Flags (internal)See Flags below
53Radio IDget3(pk+5)
83DestinationGroup or private ID (TG or radio), get3(pk+8)
114Node IDget4(pk+11)
151FlagsStart/End, Private, Slot bit
164Stream IDget4(pk+16)
2035PayloadDMR voice/data block (51 bytes total incl. header subset)
…—Total55 bytes

Flags

  • bStartStream: (flags & 0x23) == 0x21
  • bEndStream: (flags & 0x23) == 0x22
  • bPrivateCall: (flags & 0x40) != 0
  • Slot select: bit 0x80 is set for TS2 in forwarding, cleared for TS1

Login / Control Frames

TypeMagicSizeKey FieldsDescription
Login"RPTL"8NodeIDNode requests login; server replies RPTACK + salt
Auth"RPTK"40NodeID, SHA256(salt||pass)Client sends auth response; server replies RPTACK or MSTNAK
Config"RPTC"/"RPTO"≥12NodeID, INI-like payloadApplies static TGs: TS1=, TS2= (CSV)
Ping"RPTPING"11NodeIDPong with "MSTPONG" + NodeID
Ping2"FMRPING"11—Compatibility pong "FMRPONG"

Status"/STAT"5—Local status dump to 127.0.0.1:Port

Special Talkgroups

  • Unsubscribe all: 4000 (on start, server clears subscriptions)
  • Parrot: 9990 (records on start, replays on end)
  • APRS trigger: 900999 (heard report)
  • Scanner: 777 (temporary owner until traffic ends)

Packet Types & Flow

  1. Login: RPTL → RPTACK(salt)
  2. Auth: RPTK(SHA256(salt||pass)) → RPTACK or MSTNAK
  3. Config: RPTC/RPTO sets static TGs for TS1/TS2
  4. Traffic: DMRD frames → routed to all subscribers of the talkgroup (owner slot arbitration)
  5. Keepalive: RPTPING ↔ MSTPONG
  6. Logout: RPTCL

Threading & Timing

  • Time thread (time_thread_proc): increments g_tick every 50ms and g_sec each second.
  • Parrot playback (parrot_playback_thread_proc): replays buffered frames (~20ms pacing).
  • Main loop: UDP select() on server socket, dispatches to handle_rx().
  • APRS/SMS housekeeping (optional): keepalives, reconnects, and buffer emission.

Core Data Structures

  • struct node: nodeid, dmrid, salt, sockaddr, hitsec, slots[2], auth flag, timer, static TGs
  • struct slot: back-ref node, slotid, current tg, parrot buffer/seq/timestamps, optional SMS buffer
  • struct talkgroup: tg number, ownerslot, tick, linked-list subscribers
  • class memfile: in-memory expanding buffer (Read/Write/Seek)
  • class config_file: INI loader (getstring, getint)

State & Flow

Owner-slot arbitration prevents collisions: the first slot to key up on a TG takes ownership; ownership drops on end/timeout (~1.5s).

  • Joining a TG calls subscribe_to_group() (singly linked list insert).
  • Leaving TG or on control TG 4000 calls unsubscribe_from_group().
  • Scanner owner (g_scanner) follows the last active slot until end/timeout.

OpenBridge

The server now supports multiple outbound/inbound OpenBridge peers. Each enabled peer keeps its own socket, target resolution state, last RX/TX timers, optional HMAC mode, and talkgroup filtering policy.

Capabilities

  • Up to 3 peers via [OpenBridge1] .. [OpenBridge3]
  • Per-peer TG allow-list with PermitAll / PermitTGs
  • Friendly alias names surfaced in the monitor UI
  • Periodic DNS re-resolution for hostnames
  • Optional enhanced OpenBridge mode with HMAC validation/signing when built with USE_OPENSSL
  • Optional relaxed packet checks for interoperability testing

Status exposed by web API

  • Peer name / alias
  • Resolved IP and remote target
  • Network ID
  • Last RX / TX / ping ages
  • Enhanced mode flag
  • Current permit policy

Web Monitor & API

When compiled with HAVE_HTTPMODE and enabled in [Monitor], openDMR serves a static web UI and several JSON endpoints from the configured document root.

Included pages

  • index.html: dashboard with active calls, recent log data, and summary metrics
  • monitor.html: raw /STAT monitor view
  • openbridge.html: peer health and bridge status
  • register.html: self-service account registration
  • profile.html: authenticated profile and password update page

Auth model: sessions are bearer-style tokens kept in memory and refreshed on each valid request. The current implementation uses a 24-hour TTL.

The binary accepts --help. With SQLite enabled, the server auto-creates the LOG table if needed and the web UI polls the JSON endpoints roughly every 3 seconds.

Logging

Console logs use colored levels via logmsg(level, timed, ...) (Linux ANSI / Windows Console API). SQLite logging writes to a LOG table with columns DATE, RADIO, TG, TIME, SLOT, NODE, ACTIVE, CONNECT. The HTTP dashboard and monitor pages read from this same runtime state and database to render active calls, recent traffic, and OpenBridge health.

FAQ

What happens when a node sends to an unknown TG?

The server creates the talkgroup on demand. The slot can then be routed locally, forwarded to OpenBridge peers if permitted, and surfaced in the monitor/log views.

How are private calls routed?

If the destination is a radio ID within range and present in the node index, the frame is forwarded to that radio's last seen slot, with the slot bit adjusted. SMS forwarding can also allow private messages when AllowPrivate=1.

What does the embedded web server need?

Build with HAVE_HTTPMODE, set [Monitor] Enable=1, and make sure the configured Root directory contains the bundled www assets.

How does registration/profile storage work?

Registration adds entries to both the auth CSV and the configured DMRIds data file. Profile updates can change the display name and password for an authenticated DMR-ID.

How does Parrot work?

On start: create memfile buffer; on end: spawn playback thread to echo frames back at ~20ms cadence.
